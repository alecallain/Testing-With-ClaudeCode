# Architectural Patterns

## State Management: Custom Hook as Store

All app state lives in a single `useStore` hook (`src/store/useStore.ts`). There is no Context API, Redux, Zustand, or other state library. The hook:

- Fetches initial state from `GET /api/state` via TanStack Query on mount
- Subscribes to an SSE stream at `/api/events` for real-time updates from other clients
- Returns state fields and action functions as a flat object

`App.tsx` calls `useStore()` once and passes results down via props — no context providers.

## Data Flow: Prop Drilling (No Context)

State and callbacks flow from `App` -> `MonthView`/`Sidebar`/modals as explicit props. Components declare typed `Props` interfaces at the top of each file. There is no global context or event bus.

## Immutable State Updates

All store mutations call REST API endpoints via `useMutation`. SSE events then update the TanStack Query cache via `queryClient.setQueryData`. Updates spread the previous state and replace only the changed collection:

```
patch((prev) => ({ ...prev, chores: [...prev.chores, newChore] }))
```

Cascading effects are handled server-side (e.g., removing a member also filters them from all chore `assigneeIds` arrays) and broadcast as a single SSE event with the updated data.

## Entity ID Pattern

Entities (`Chore`, `Member`, `Completion`) are created with `Omit<Entity, 'id'>` as the input type; `crypto.randomUUID()` assigns the `id` inside the store mutation before sending to the API. IDs are never generated by the caller.

## Controlled Modal Forms

Modals (`ChoreModal`, `TeamModal`) manage their own local form state. A `useEffect` keyed on `open` resets the form from the `initial` prop each time the modal opens (`src/components/Modals/ChoreModal.tsx`). The modal calls `onSave(formData)` and `onClose()` on submit; the parent decides whether to call `addChore` or `updateChore`.

## Derived Data via useMemo

`MonthView` (`src/components/CalendarView/MonthView.tsx`) computes two expensive derived values with `useMemo`:

1. `days` — the full calendar grid (6 weeks of `Date` objects) for the current month
2. `occurrenceMap` — a `Record<dateStr, ChoreOccurrence[]>` built by running `getOccurrencesInRange` over every chore for the visible range

This avoids recomputing the calendar grid on every render.

## Date Conventions

- Dates are stored as `"YYYY-MM-DD"` strings in all entities (`startDate`, `endDate`, `Completion.date`)
- `date-fns` functions (`parseISO`, `format`, etc.) are used for all parsing and formatting — never `new Date(string)` directly
- `date-fns` is also used for all calendar math (start/end of month/week, day iteration, month navigation)

## Recurrence Logic

All recurrence calculation is isolated in `src/utils/recurrence.ts` (`getOccurrencesInRange`). It handles `none | daily | weekly | monthly` via a `switch` and returns `string[]` of matching dates. Monthly recurrence clamps to end-of-month (e.g., day 31 in February becomes Feb 28/29).

## Color System

Colors are hex strings (not Tailwind class names) for chore chips and member avatars. Palette constants live in `src/utils/colors.ts`. `Member.color` is stored as a hex string. Use inline styles for all dynamic colors — never construct dynamic Tailwind class names.

## Multi-Assignee Pattern

`Chore.assigneeIds` is a `string[]` of up to 4 member IDs (replaces the old single `assigneeId: string | null`).

- **Resolution**: `MonthView` and `DayCell` resolve `assigneeIds` to `Member[]` by filtering the `members` array, then pass the resolved array as `assignees: Member[]` to `ChoreChip`. Never pass raw IDs to presentational components.
- **Quick add/remove**: `ChoreChip` receives `assignees`, `allMembers`, and `onToggleAssignee(memberId)`. The popover shows current assignees with × remove buttons and available members with + add buttons. The popover stays open after each toggle (unlike `onToggleDone` which closes it).
- **Max-4 enforcement**: layered at the store action (`toggleAssignee` checks `length < 4`), the modal UI (pill buttons disabled at max), and the popover (add section hidden when full).
- **Cascading removal**: when a member is deleted, `server/db.ts:deleteMember` reads all chores and filters the member's ID out of every `assigneeIds` array, then broadcasts the updated chore list via SSE.
- **SQLite storage**: `assigneeIds` is stored as a JSON string (e.g., `'["id1","id2"]'`) in the `chores` table. `rowToChore` parses it with `JSON.parse`; `insertChore`/`updateChore` serialize with `JSON.stringify`.
