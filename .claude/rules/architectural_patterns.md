# Architectural Patterns

## State Management: Custom Hook as Store

All app state lives in a single `useStore` hook (`src/store/useStore.ts:20`). There is no Context API, Redux, Zustand, or other state library. The hook:

- Initializes from `localStorage` on mount (`loadState`, line 6)
- Persists every update to `localStorage` via a shared `update` callback (line 23)
- Returns state fields and action functions as a flat object

`App.tsx` calls `useStore()` once and passes results down via props — no context providers.

## Data Flow: Prop Drilling (No Context)

State and callbacks flow from `App` -> `MonthView`/`Sidebar`/modals as explicit props. Components declare typed `Props` interfaces at the top of each file. There is no global context or event bus.

## Immutable State Updates

All store mutations use the functional `setState(prev => next)` form (`src/store/useStore.ts:24`). Updates spread the previous state and replace only the changed collection:

```
update((prev) => ({ ...prev, chores: [...prev.chores, newChore] }))
```

Cascading effects happen in a single update (e.g., removing a member also unassigns their chores and `removeChore` also deletes related completions).

## Entity ID Pattern

Entities (`Chore`, `Member`, `Completion`) are created with `Omit<Entity, 'id'>` as the input type; `crypto.randomUUID()` assigns the `id` inside the store action. IDs are never generated by the caller.

- `src/store/useStore.ts:37` — `addMember`
- `src/store/useStore.ts:63` — `addChore`
- `src/store/useStore.ts:104` — `toggleCompletion`

## Controlled Modal Forms

Modals (`ChoreModal`, `TeamModal`) manage their own local form state. A `useEffect` keyed on `open` resets the form from the `initial` prop each time the modal opens (`src/components/Modals/ChoreModal.tsx:31`). The modal calls `onSave(formData)` and `onClose()` on submit; the parent decides whether to call `addChore` or `updateChore`.

## Derived Data via useMemo

`MonthView` (`src/components/CalendarView/MonthView.tsx`) computes two expensive derived values with `useMemo`:

1. `days` — the full calendar grid (6 weeks of `Date` objects) for the current month
2. `occurrenceMap` — a `Record<dateStr, ChoreOccurrence[]>` built by running `getOccurrencesInRange` over every chore for the visible range

This avoids recomputing the calendar grid on every render.

## Date Conventions

- Dates are stored as `"YYYY-MM-DD"` strings in all entities (`startDate`, `endDate`, `Completion.date`)
- `date-fns` functions (`parseISO`, `format`, etc.) are used for all parsing and formatting — never `new Date(string)` directly
- `date-fns` is also used for all calendar math (start/end of month/week, day iteration, month navigation)

## Recurrence Logic

All recurrence calculation is isolated in `src/utils/recurrence.ts:19` (`getOccurrencesInRange`). It handles `none | daily | weekly | monthly` via a `switch` and returns `string[]` of matching dates. Monthly recurrence clamps to end-of-month (e.g., day 31 in February becomes Feb 28/29).

## Color System

Colors are hex strings (not Tailwind class names) for chore chips and member avatars. Palette constants live in `src/utils/colors.ts`. `Member.color` is also stored as a hex string despite an old comment in `src/types.ts:12` suggesting Tailwind classes — the actual values used are hex.
